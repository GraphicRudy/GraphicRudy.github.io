04

Connect your data.

Home is where you pick a source. Connected sources stay ready across sessions — authenticate once, Karl remembers until credentials expire.

Steps

Open Home

Sidebar → Home. Connected sources are cards at top; available ones below.

Click "Add [source]"

Connector catalog shows every source type your tenant has enabled.

Authenticate

OAuth for cloud warehouses, username/password for databases, file upload for CSV/XLSX. Credentials stored server-side via encrypted vault.

Verify connection

Karl runs schema discovery. You'll see table count and a green "Connected" badge.

Start querying

Click the source card. Source chip stays persistent in the top bar — you always know which source you're querying.

lightbulb
Switching sources preserves history

New in v0.2: switching sources archives your current conversation, doesn't delete it. Return from the Sessions panel.

05

Ask questions.

The primary interface. Type a question in plain English, get an answer with the SQL, review, refine.

Anatomy of a Karl response

Every answer has four parts — new in v0.2. v0.1 returned just the result table.

  1. Interpretation summary — which tables, how ambiguous terms were resolved
  2. Confidence signal — High/Medium/Low. Low is a cue to verify
  3. View SQL toggle — reveals generated SQL. Essential for finance/compliance
  4. Result + follow-ups — the data, plus suggested follow-up questions

Tips for effective queries

  • Be specific about time — "Q2 2026" or "last 90 days" beats "last quarter"
  • Use your org's terms — if your team calls it "ACV," say "ACV." Karl learns via Training Data.
  • Follow up, don't restart — "break that down by region" is faster than a new question
  • Always check SQL on high-stakes queries — finance, audit, board reporting
check_circle
Conversation persistence

Conversations are saved per-user, per-source. Other users can't see your conversations unless you explicitly share.

When things go wrong.

Karl has documented error states now — v0.1 surfaced errors but didn't explain them. The top eight and how to recover.

Column not found

E-QRY-001
Karl referenced a column not in the schema.
Fix
  • Use exact column names
  • Add a training pair (Ch 07)

Connection lost

E-SRC-002
Credentials expired or network down.
Fix
  • Home → source card → re-authenticate
  • Contact admin if persistent

Query timeout

E-QRY-008
Ran longer than 60s default.
Fix
  • Add filters (date range, LIMIT)
  • Ask admin to raise timeout

Permission denied

E-PRM-003
Karl's service account lacks SELECT on the table.
Fix
  • Ask data team to grant SELECT
  • Query a different table

Ambiguous term

E-QRY-004
Term maps to multiple fields/tables.
Fix
  • Answer Karl's clarification
  • Add training pair for disambiguation

Import format invalid

E-IMP-005
CSV doesn't match expected format.
Fix
  • See Chapter 08 for spec
  • Download template from Training Data

Invalid SQL in pair

E-TRN-006
Training pair SQL fails to parse or execute.
Fix
  • Test SQL in a query tool first
  • Verify column names match schema

Rate limit exceeded

E-SYS-007
Tenant per-minute query cap hit.
Fix
  • Wait 60s and retry
  • Ask admin to raise tier limit
warning
Errors are diagnostic, not terminal

Every error screen has "Try rephrasing," "Back to conversation," and where relevant "Add training pair" pre-filled with what Karl thinks the fix is.

07

Training Data — teach Karl your language.

A training pair is a plain-English description matched to the SQL that answers it. More pairs = better domain-specific performance.

sync
Sync timing: immediate

New in v0.2: training pairs apply to the next query — no batch sync. ~5 seconds from save to availability.

Adding a single pair

Open Training Data

Sidebar → Training Data. Existing pairs shown with status.

Click "Add training pair"

Modal opens with Natural Language + SQL Query fields.

Enter the description

Write how a user would actually ask. Example: Total sales by region for this year

Enter the SQL

Karl validates syntax and dry-runs against your schema. Invalid pairs never apply.

Save

~2 seconds to validate. If passed, applies on next query. If failed, inline error with fix suggestion.

lightbulb
What makes a good pair

Do: use exact team phrasing. Cover variations. Keep SQL tested. Don't: submit draft SQL. Don't duplicate Karl's built-in capabilities.

Bulk import

For migrating existing libraries or adding 20+ pairs. Format spec in Chapter 08.

08
New in v0.2

CSV / XLSX import format.

v0.1 said bulk import existed but didn't specify format. Here's the spec. Template downloadable from Training Data.

Required columns

Column A: natural_language_description // required, string, 1–500 chars Column B: sql_query // required, string, 1–8000 chars Column C: domain_tag // optional: finance | operations | hr | marketing Column D: owner_email // optional, defaults to uploader Column E: notes // optional, up to 1000 chars

Constraints

  • Max file size: 10 MB
  • Max rows: 2,000 pairs per import
  • Header row: required on row 1 with exact column names
  • Encoding: UTF-8 for CSV
  • Escape rules: SQL with commas wrapped in double-quotes. Double-quotes escaped as "" per RFC 4180.

Example row

"Total revenue this quarter","SELECT SUM(revenue) FROM fact_orders WHERE quarter = ""Q2-2026""",finance,raj@company.com,"Primary revenue definition"
warning
Validation is per-row

Failed rows rejected individually — import continues for valid rows. You'll see a validation report with per-row errors. Fix and re-upload rejected rows.

09

Knowledge Guide — shape how Karl thinks.

Training Data teaches terminology. The Knowledge Guide teaches reasoning — how to handle ambiguity, which assumptions to make.

Two layers

LayerScopeExample
Critical
● ADMIN ONLY		Global. Apply to every response."Default to fiscal calendar when time is ambiguous." "Redact PII unless explicitly requested."
Auxiliary
● POWER + ADMIN		Domain-specific. Layer on top of Critical."For Finance: use recognized_revenue for 'revenue' questions." "For HR: anonymize performance data."

How activation works v0.2 clarification

v0.1 said Karl "detects" domain but not how. In v0.2, activation is explicit:

  1. Domain tagging — each Auxiliary set has activation terms (e.g., Finance activates on "revenue," "MRR," "quarter")
  2. Classifier fallback — if no term match, LLM classifier picks the best-fit set
  3. Manual override — pin a domain for a session ("Query as Finance")
dangerous
Test Critical changes before publishing

Critical Instructions apply to every response. Use Test in sandbox (v0.2 feature) before saving. All changes versioned — rollback is one click.

Version history

Every save creates a version with author, timestamp, diff. Rollback is one click. v0.1 had no rollback — the #1 admin complaint.

10

Telemetry Insights (Admin only).

View of how Karl is used across the organization. Use it for governance, adoption tracking, capacity planning.

Metrics

  • Query volume — daily/weekly/monthly trends
  • Active users — count per period
  • Success rate — % returning a result without error
  • Response time — p50, p95, p99
  • Query type distribution — % by domain
  • Employee usage — per-user volume, success rate, top domain

Filters (new in v0.2)

  • Period — 7/30 days, quarter, custom
  • Team — filter by org team
  • Data source — isolate one source
  • Domain — focus on one business function
privacy_tip
Use per-user data responsibly

Employee breakdown includes individual-level data. Review your privacy policies before exporting. Intended for adoption analysis, not performance evaluation.

A

Glossary.

Product-specific terminology. v0.2 surfaces these at point of need via inline tooltips.

Training Pair
Matched natural-language description + SQL query. Teaches Karl how your org asks questions.
Example: "monthly recurring revenue" → SELECT SUM(monthly_value) FROM subscriptions WHERE status='active'
Critical Instructions
Foundational rules applied globally to every response. Admin-configured.
Example: "Default to fiscal calendar when time is ambiguous."
Auxiliary Instructions
Domain-specific rules layered on top of Critical. Scoped to Finance, Operations, HR, etc.
Example: Finance Auxiliary defines "revenue" as recognized revenue.
Tenant
Organizational unit within Karl. All users, sources, configuration live within a tenant.
Telemetry
System-generated data about Karl usage — volumes, types, success rates. Admin-only.
Data Source
External database, warehouse, file, or API Karl queries on your behalf.
Natural Language Query (NLQ)
Question asked in plain English. Karl translates into SQL.
Confidence Signal
High/Medium/Low label on each response. Low = review the SQL.
Activation term
Keyword that triggers a specific Auxiliary Instruction set. Example: "MRR" activates Finance.
